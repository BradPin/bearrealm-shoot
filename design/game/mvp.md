# MVP1 範圍定義（Bearrealm Shoot）

> **狀態**：已與 Producer 對齊（單人驗證優先、架構預留多人；遺物採過渡方案 D）。  
> **用途**：新對話與開發收斂範圍的依據；細節仍以 `glossary.md` 與各系統企劃為準。

---

## 1. 一句話目標

在 **熊域世界觀 + 類怪物彈珠式回合戰鬥** 的前提下，做出 **可從頭到尾持續推關、規則一致且可重複遊玩** 的 MVP1；遊玩情境以 **單一真人玩家** 為主，但 **程式架構需利於日後擴充為真人多人**。

---

## 2. MVP1 遊玩模式（對玩家）

- **唯一主力情境**：**1 名真人玩家 + 3 名 AI 代管**（固定 4 顆彈珠的操作單位）。
- **不做為 MVP1 驗收條件**：2～4 真人連線、同步、連線品質、斷線重連、配對大廳等（見第 6 節 Out of Scope）。
- **設計假設（供日後企劃補文）**：無論幾位真人，**空位皆由 AI 代管**；單人模式僅是「真人數 = 1」且 **無需跨端同步遊戲狀態** 的特殊情況。

---

## 3. 架構原則（程式／系統）

- **MVP1 實作與測試重心**：本機單人流程（無連線同步需求）。
- **擴充預留**：核心戰鬥與關卡進行方式應能以 **「可抽換的輸入／權威與同步層」** 的思路設計，避免把「只有本機才成立」的邏輯寫死在 UI 或單一全域單例中；**具體網路模型與狀態同步策略**由 Producer 於企劃／技術備忘中補齊，MVP1 不強制實作。
- **與企劃的關係**：單人與多人的差異、AI 代管規則、未來同步邊界 → **建議獨立成企劃文件**（見第 8 節），避免僅存在於對話或程式註解。

---

## 4. 遺物編織（MVP1 策略）

- **主方案（D）**：**假 UI／極簡 UI + 真資料**。  
  - 遺物相關 **資料結構與術語** 應與長期設計對齊（見 `system/relic-weave.md`、`glossary.md`），避免 MVP1 後整表作廢。  
  - 玩家-facing 的編織介面 **不要求完整功能**，以能 **推進戰役、驗證戰鬥與掉落循環** 為限。
- **備援（若實作或隨機銜接過重）**：**事先定義一條固定遺物鏈**；**每一關**以簡單規則從預設集合中 **隨機啟用／開放一個節點或效果**（細節由 Producer 在遺物企劃中定死），確保 MVP1 **不重做 schema** 又能通關。

---

## 5. 戰役與關卡內容（MVP1）

- **流程**：正常戰役流程，可 **持續往後打**（進關 → 戰鬥 → 結算 → 下一關），直至企劃為 MVP1 定義的結束條件（若尚未定義，預設為「可無限推進＋內容重複可接受」）。
- **關卡類型**：**小／中／大** 三類；MVP1 每類 **約 1～3 張** 地圖即可，推關時 **重複同一關卡圖** 可接受。
- **內容量**：以 Producer 列出的最小清單為準，**總量偏少** 但須足以 **跑完整迴圈**（戰鬥、勝負、進關、遺物過渡流程）。
- **勝利／失敗條件**：以 **關卡 schema 欄位** 定義（見 `system/level.md` 與 Producer 補充之 schema 說明）；MVP1 可先只支援 **一種或少數幾種** 條件類型，其餘標註 Phase 2。

---

## 6. In Scope / Out of Scope

### 6.1 In Scope（MVP1 應完成）

- 1 真人 + 3 AI 的 **可玩通關循環**（戰鬥核心 + 關卡推進 + 與遺物過渡方案銜接）。
- **關卡資料驅動**：含 **勝利條件等** 由 schema 表達（實作可逐步）。
- **遺物**：長期相容之 **資料形狀** + **D 方案或備援固定鏈** 其一（由實作與企劃共同敲定）。
- **小／中／大** 類型關卡之 **最小組合**（可重複使用同一資產）。

### 6.2 Out of Scope（MVP1 不驗收；可留擴充點）

- 真人多人連線、狀態同步、延遲補償、斷線重連、公開配對。
- **完整** 遺物編織 UI 與完整隨機掉落經濟（除非降級為備援方案並寫入企劃）。
- 完整城鎮／局外 meta、敘事演出、大量美術與音效產能（僅需支撐可玩迴圈之最小集）。

---

## 7. 驗收標準（建議）

- 單一建置版本下，**無需其他真人玩家**，即可從戰役入口玩到 **多次關卡推進** 且 **戰鬥勝負與進關邏輯一致、無阻斷性 bug**。
- **關卡勝負** 依 schema 設定可重現（改欄位能改變結果，而非寫死僅一種關）。
- **遺物**：至少一條可跑通的 **資料 + 流程**（D 或固定鏈備援），並與 `glossary` 用語一致。

---

## 8. Producer 須準備之企劃文件（MVP1 前建議「寫滿可執行」）

| 優先序 | 文件（建議路徑／現有檔） | MVP1 應寫到的深度 |
|--------|--------------------------|-------------------|
| P0 | `glossary.md`（術語維護） | 任何新名詞先進表；MVP1 新增之關卡／遺物用語需一致。 |
| P0 | `system/combat.md` | 回合流程、同時彈射、技能自動釋放順序、勝負檢查時機、邊界情況（全靜止、同時觸發等）**可給程式實作** 的說明。 |
| P0 | `system/level.md` + **關卡 schema 說明**（可同檔或 `level_schema.md`） | **欄位定義**（含 **勝利／失敗條件** 類型與參數）、小／中／大類型意義、MVP1 最小欄位集。 |
| P0 | `system/marble.md` | 單一彈珠屬性、魔力取得方式在 MVP1 的取捨、與 AI 代管分工（若影響規則）。 |
| P0 | **單人／多人與 AI 代管規格**（新建建議：`system/session_multiplayer.md`） | MVP1 僅 **1+3**；未來多人時 **席位、操作權、與 AI 補位**；**何時不需同步**（單機）與 **未來需同步的邊界**（概念層即可）。 |
| P1 | `system/relic-weave.md` + **MVP1 遺物流程附錄**（可同檔加一小節「MVP1」） | 採 **D** 或 **固定鏈 + 每關隨機開放** 的**明確規則**；與 `glossary` 能量／Inactive 等一致；若簡化需標「僅 MVP1」。 |
| P1 | **MVP1 內容清單**（新建建議：`mvp1_content_backlog.md`） | 關卡 ID、敵人／技能模板、遺物 ID **最小表**；數量可少，但須能跑通。 |
| P1 | **戰役流程與結束條件**（新建或併入 `基礎設定.md`） | 從進戰役到結算、下一關；MVP1 是否無限推進或有限章節。 |
| P2 | **關卡數值縮放**（新建建議：`system/progression_scaling.md`） | 隨關卡加深之公式或規則；若 MVP1 用常數或簡表，需註明並可之後替換。 |
| P2 | `世界觀.md` / `基礎設定.md` | MVP1 可不深寫，但 **不得與戰鬥規則矛盾**。 |
| P2 | `vibe.md` | 畫面與氛圍參考；Agent 產出資產時對齊。 |

**Agent／技能側產出**（非上表「Producer 必寫滿」）：場景結構、資源管線、UI 佈局實作、程式架構與測試等，依 `.agent/skills/` 分流；**以 Producer 文件為準**，歧義時回頭增補企劃而非自行改玩法。
